Definitions and Notations

Canonical

The term canonical is used as a synonym of salient. So canonical can refer to any of the following.

  1. the most noticiable
  2. the most important
  3. the primary
  4. the most obvious
  5. the most obiouis and trivial

In the surrogate math section, the canonical mappings are not obvious, even though they are obtained by "following your nose", so canonical there means the most important. Every where else it means obvious and trivial. When that's the meaning, details will not necessarily be provided.

Function Notation

The notatoin x ∈ X means x is an element of of the set X. Let's look at the notation. m:X → Y, m(x) = x + 1 (if x ∈ X) This means that m is a map from the set X to the set Y. After this may come the rule or formula for the map, written in green. Then after this may come a description of the elements for which the formula holds, written in maroon. Also X is called the domain of the map.

  1. if x ∈ X, then m(x) is the image of x.
  2. m(X) denotes the range or image of X. It's the set of all elements y of Y for which there is an element of x of X for which m(x) = y. In set notation m(X) = {m(x): x ∈ X}.

  3. If m is one-one then there is an inverse map from the maps image back to it's domain.

    m-1:m(X) → X, m-1(m(x)) = x, (if x ∈ X)

    It is also true that

    m(m-1(y)) = y, (if y ∈ m(X))
  4. If m is one to one and onto, then m is an equivalence. Also if m is a canonicl map, then m is a canonical equivalence.

Ranges

The delimiters [] are used to dentote ranges (of integers)

[2,6] is the range of integers from 2 to 6 inclusive. It's the set {2,3,4,5,6}.

Sequence Spaces

The delimiters <> are used to denote sequences.

<A, D, F> is a sequence of three letters.

A pair is a seqence of two. X x Y is the set of all pairs where the first entry is an element of X, and the second entry is an element of Y. X x Y is also called the cross product of X and Y.

If X is a set then Xs denotes the set of all sequences whose entries are menbers of X. It is implility understood that Xs has the natural operator of concatenation, which is denoted with the dot ·. Concatenation is an associative operation.

Sequential Maps

A map m:Xs → Ys is a sequential map if it preserves concatenation. That is m(s·t) = m(s)·m(t) (whenever s, t ∈ Xs)

Sequential Extensions

If m:X → Ys is a map then there is a canonical sequential extension ms:Xs → Ys, ms(<x1,...,xn>) = m(x1)···m(xn) (if x1,...,xn ∈ X)

If m:X → Y is a map then there is a canonical sequential extension ms:Xs → Ys, ms(<x1,...,xn>) = <m(x1),...,m(xn)> (if x1,...,xn ∈ X)

Bit Stream

A bit stream is a sequence of bits in computer storage. Commas are not used to separate the entries. Spaces may be used for readability

The concatentation of the three bit streams 1011, 11001, and 0 is 10 1111 0010.

20 and 10 bit Ranges = UTF-16 Surrogate Math

We get a head start. You need to know nada about Unicode and UTF-16 to understand this section.

We show how to canonically decompose an astral number into a high surrogate number and a low surrogate number pair.

We also show how to canonically compose a high surrogate number and low surrogate number pair into an astral number.

This means there is a canonical equivalence between astral numbers and high surrogate numbers/low surrogate number pairs.

If you are good with binary and hexidecimal numbers you can very quickly come up with the surrogate formulas from "Wikipedia UTF-16: Examples". The following is the long approach since it gives all the simple details.

The quick summary is that given any 20 bit range of numbers and any two 10 bit range of numbers, any number a in the 20 bit range detrmines a pair of numbers <h,l> where h lies in the first 10 bit range and l lies in the second 10 bit range. This map is reversible, in that a determines <h,l> and <h,l> determines a.

In our particular example, the astral numbers form the 20 bit range. The high surrogates and low surrogates form the two 10 bit ranges.

The 20 bit numbers are all the non-negative integers that can be written with exactly 20 binary bits with possible leading zeros. Likewise for 10 bit numbers.

20 bit numbers = the range [0, 1...1220-times] = [0x0, 0xFFFFF]
(All integers between 0x0 and 0xFFFFF inclusive)

10 bit numbers = the range [0, 1...1210-times] = [0x0, 0x3FFF]
(All integers between 0x0 and 0x3FFF inclusive)

The calculations for the diagram are

[0, 1...1220-times] = 1111 1111 1111 1111 11112 = FFFF16= 0xFFFF.
[0, 1...1210-times] = 11 1111 11112 = 3FFF16= 0x3FFF.
0x10FFFF - 0x10000 = 0xFFFF.
The high 10 bits of a 20 bit number x is y = x >> 10.
The low 10 bits of x is z = 11 1111 11112 & x = 0x3FF & x .
0x3FF + 0xD800 = 0xDBFF.
0x3FF + 0xDC00 = 0xDFFF.

We also need to know that if y and z are the high and low 10 bits of x then they can be snapped together via x = y << 10 + z to get x.

Canonical Decompositon of an Astral Number into a High and Low Surrogate Pair

Following the diagram yields the following formula.

a → <h,l>
h = ((a - 0x10000) >>10) + 0xD800 and  l = (a - 0x10000) & 0x3FFF + 0xDC00

You could also follow the diagram bottoms up and use substitution.

h = y + 0xD800
  = (x >> 10) + 0xD800
  = ((a - 0x10000) >> 10) + 0xD800;

l = z + 0xD800
 = (x & 0x3FF) + 0xDC00
 = ((a - 0x10000) & 0x3FF) + 0xDC00

Canonical Composition of a High and Low Surrogate Pair into an Astral Number

To get the reverse formula, we just follow the diagram top down, and use substitution by reversing each step individually.

a = x + 0x10000
 =(y << 10 + z) + 0x10000
 = ((h - 0xD800) << 10 + (l - 0xDC00)) + 0x10000

So the reverse forumula follows.

a ← <h,l>
a = ((h - 0xD800) << 10) + (l - 0xDC00) + 0x10000

We have encoded an astral number into its high and low surrogates (canonical decomposition). Its high and low surrogates can be used to obtain the original astral number (cononical composition). We used different functional notations than in the introduction. Let's make amends now.

The map

𝔸: Astral Numbers → High Surrogtes x Low Surrogates
𝔸(a) = <((a - 0x10000) >> 10) + 0xD800, ((a - 0x10000) & 0x3FF) + 0xDC00> (if a is an astral number)

is a canonical equivalence (is one to one and onto). It's inverse is

𝔸-1: High Surrogtes x Low Surrogates →Astral Numbers
𝔸-1(<h,l>) = ((h - 0xD800) << 10) + (l - 0xDC00) + 0x10000 (if h is a high surrogate and l is a low surrogate)

Unicode Space

A unicode code point is any integer in the range [0, 1114111] = [0x0,0x10FFFF]. The code point is usually written in decimal or hexidecimal. Every code point determines a unicode number obtained by concatenating the abstract unit U with the code point's hexidecimal digits through addition.

Thus the code point 193 = 0xC1 determines the unicode number U + C1, and the code point of U + C1 is 0xC1 = 193.

The range of unicode space is [U + 0, U + 10FFFF] which corresponds to the range [0, 1114111] = [0x0,0x10FFFF] of code points.

Since there is a canonical/trivial one-one correspondence between unicode numbers and (unicode) code points, the notions of unicode number and code point may be conflated without any misunderstanding.

For example, even expert source material will call the unicode number U + FECD a code point. Conversely, the code point 0xFECD may be referred to as a Unicode number.

Since code points are numbers, we know what the surrogate code points are, what the high surrogate code points are, what the low surrogate code points are, and what the astral code points are. The non astral code points are the the BMP code points.

Machine Encodings of scalar value sequences

Definitions to follow are mildly different than might be found elsewhere, but only because total accuracy is desired. For example, a sequence of code units is not the same as their concatenation. If we mean concatenation (because the specification means it), we say it, not imply it.

There are exactly three machine encodings of Unicode space that are currently part of the Unicode standard, UTF-8, UTF-16, and UTF-32. By machine encoding, it is meant that scalar values are concretely encoded as bit streams.

In each UTF encoding form, every scalar value is concretely encoded as the concatenation of a sequence of code units, where a code unit is a bit stream of fixed length.

A UTF encoding form does not encode surrogate code points. You could but you can't. Well you probably can, but it's illegal and you would end up in the gulag run by the Consortium's Unicode Technical Committee.

  1. A UTF-8 code unit is an 8 bit stream.
    • A scalar value is concretely encoded as the concatenation of a sequence of one to four code units. This is a stream of 8, 16, 24, or 32 bits.
  2. A UTF-16 code unit is a 16 bit stream.
    • A scalar value is conceretely encoded as one code unit or the concatenation of a sequence of two code units. This is a stream of 16 or 32 bits.
  3. A UTF-32 code unit is a 32 bit stream.
    • A scalar value is concretely encoded by one code unit, which is a stream of 32 bits.

UTF encodings of scalar values canonically extend to UTF encodings of scalar value sequences. The UTF encoding of a scalar value sequence is the result of concatenating the UTF encodings of each scalar value in the sequence.

A sequence of code units such as (1) <0x0041, 0xD835, 0xDD61>
is canonically equivalent to a sequnce of binary bits. Merely convert each code unit into a 16 bit binary sequence and concatentate the results to obtain

(2) 0000 0000 0002 0001 1110 1000 0011 0101 1110 1110 1110 0110 0001.
Since (1) and (2) are canonically equivalent we can use (1) as shorthand for (2).

UTF-16 Encoding

UTF-16 encoding is defined as follows

  1. The UTF-16 encoding of a BMP scalar value is the canonically equivalent code unit, which is a 16 bit stream.
  2. The UTF-16 encoding of an astral scalar value is obtained by computing its canonical decomposition. Then each member of this sequence is replaced by their canonically equivalent 16 bit stream to obtain a sequence of two code untis. Then these two code units are concatenated to obtain a 32 bit binary stream as the encoding of the astral scalar value.

It can be deduced that there are no restrictions on a UTF-16 code unit other than that it consists of 16 bits. Thus there is a canonical equivalence between code units and the numbers in the range [0x0000, 0xFFFF], which is the range of BMP code points. Also there is a canonical equivalence between code units and hexidecimal representations with exactly four hexidecimal digits, with possible leading zeros to be implicitly understood.

Through the former canonical equivlannce we know what the surrogate code units are, and in particular the high surrogate code units and the low surrogate code untis. Merely convert the 16 bit code unit to a number. If the number is a surrogate number then the code unit is a surrogate code unit. Likewise for high and low surrogate code units.

Surrogate code units are allowed, and are in fact necessary in UTF-16 encoding. Keep in mind the nuanced contrast that UTF-16 does not encode surrogate code points.

    • A stand alone surrogate code point is either a high surrogate code point in a coded character sequence that is not followed by a low surrogate code point, or a low surrogate code point that is not preceded by a high surrogate code point. A coded character sequence containing a stand alone surrogate can not be a UTF-16 coded character sequence.
    • The Consortium does not have an equivalent concpet. Another missed opportunity to define something useful.
      • If you know something about Unicode, don't confuse UTF-16 coded character sequence with well formed UTF-16 Unicode string, which is an analoguos concept, not the same concept.
  2. A UTF-16 Unicode string is a sequence of UTF-16 code units. It is well formed if surrogate code units come in pairs (no stand alone surrogate code units).
    • Elsewhere in the specification, it's pretty clear that a UTF-16 Unicode string is actually meant to be a bit stream. Thus in the above definition what is really meant is that a UTF-16 Unicode string is the concatenation of a sequence of UTF-16 code units. Rather than modifying the official defintion above, we will let context be the guide. If context doesn't distiguish between the two, take whichever one you want because the two definitions are canonically equivalent.

By canonical extension, UTF-16 concretely encodes every scalar value sequence into a well formed UTF-16 Unicode string.

Consequently, UTF-16 concretely encodes every UTF-16 coded character sequence into a well formed UTF-16 Unicode string.

By canonical extension, UTF-16 concretely encodes every abstract character sequence into a well formed UTF-16 Unicode string.

Structure of Unicode Space

A unicode code point is any integer in the range [0, 1114111] = [0x0,0x10FFFF]. The code point is usually written in decimal or hexidecimal. Every code point determines a unicode number obtained by concatenating the abstract unit U with the code point's hexidecimal digits through addition.

Thus the code point 193 = 0xC1 determines the unicode number U + C1, and the code point of U + C1 is 0xC1 = 193.

The range of unicode space is [U + 0, U + 10FFFF] which corresponds to the range [0, 1114111] = [0x0,0x10FFFF] of code points.

Since there is a trivial one-one correspondence between unicode numbers and (unicode) code points, the notions of unicode number and code point may, and will be, conflated without any misunderstanding.

For example, even expert source material will call the unicode number U + FECD a code point. The author will use the terms interchangebly.

Unicode space is a virtual itermediary encoding space between abstract characetrs (next section) and bytes of computer storage. An abstract character is virtually encoded as a code point or sequence of code points, which themsevles are concretely encoded into bytes of computer storage through one of the encoding forms UTF-8, UTF-16, or UTF-32 to otain a sequence of bytes that concretely encodes the abstract character in computer storage.

UTF-16

There are exactly three encodings that are currently part of the Unicode standard, UTF-8, UTF-16, and UTF-32.

In each UTF encoding form, every scalar valueUnicode number is encoded or mapped to a sequence of code units, where a code unit is a sequence of bits of fixed length. The mapped to sequence of code units is the encoding of the Unicode number. It represents a sequence of bits in computer storage.

"MSDN: Code Units"
  1. A code unit in UTF-8 consists of 8 bits. (a code point is encoded by one to four code units)
  2. A code unit in UTF-16 consists of 16 bits. (a code point is encoded by one or two code units)
  3. A code unit in UTF-32 consists of 32 bits. (a code point is encoded by one code unit)

In the HTML document's meta tag, any of the three machine encodings may be specified, and will apply to HTML. I found no documentation that says it applies to JavaScript. However, it probably does since we shall later see that any of the three machine encodings may applly to JavaScript. It's just that magic must be performed behind the scenes to make it appear that JavaScript is using UTF-16 even if it isn't

W3.org has a strong opinion of which machine encoding to specify in the meta tag. (You will have to scroll down the page to find the quote).

"W3: HTML encoding"
You should always use the UTF-8 character encoding unless you shouldn't. (Remember that this means you also need to save your content as UTF-8.) See what you should consider if you really cannot use UTF-8.

Stand alone surrogates are ill formed. Given a string of UTF-16 codeponts, the ill formed stand alone surrogates are obtained by reading left to right. If a high surrogate is not followed by a low surrogate, then the high surrogate is ill formed. The whole string is then ill formed. If a low surrogate is encountered, not preceded by a high surrogate, then the low surrogate is ill formed and the whole string is ill formed.

If an UTF-16 Unicod string is specified as being well formed then it is highly illegal/non coformant for the string to be ill formed, of course because that would be an outright lie. If the UTF-16 string is not specified as being well formed, then it is not illegal for it to be ill foremd. In fact it is expected that the string might be ill formed. For example, strings in Java, C#, or ECMAScript are Unicode 16-bit strings, but are not necessarily well-formed UTF-16 sequences.

JavaScript encodes Unicode numbers at the language level. Like the UTF-16 machine encodings, a Unicode number still maps to a sequence of one or two code units. But at the language level, the meaning of code unit must change. A code unit refers to a 16 bit logical value of text. Read on.

Now even though UTF-8 is most commonly used and is the recommended machine encoding, it's ironic that only UTF-16 corresponds extremely well with JavaScript language encodings. You can directly use big endian UTF-16 encodings in JavaScript. The UTF-16 encoding of certain Unicode numbers into surrogate pairs translates directly into JavaScript. See Wikipedia: UTF-16, which you probably should read.

So simple facts about UTF-16 encoding can be used as ratinoliaztions for JavaScript language encodings, even if UTF-8 or UTF-32 machine encodings are used. Further definitive evidence follows.

"O'Reilly--Speaking JavaScript--An In-Depth Guide for Programmers: Source Code Internally"

Internally, JavaScript source code is treated as a sequence of UTF-16 code units. According to Section 6 of the EMCAScript specification:

"EMAC Specification: Section 6"
ECMAScript source text is represented as a sequence of characters in the Unicode character encoding, version 3.0 or later. [...] ECMAScript source text is assumed to be a sequence of 16-bit code units for the purposes of this specification. Characters and Code units

We faitfully follow section 6 of the EMAC specification to narrown down the meaning of character in JavaScript. At the language level, a code unit and a character both refer to 16-bit logical values of text.

  1. '\u223C' is a JavaScript encoding of U + 223C. If you count the number of logical bits its 4 x 4 = 16. Thus '\u223C' is a JavaScript character.
  2. Like wise '\x0F' which can be written as '\u000F' counts as 16 bits, whence '\x0F' is a JavaScript character.
  3. '\u{1D566}' counts as 20 logical bits, whence '\u{1D566}' is not a JavaScript character.
  4. A JavaScript string consists of exactly one character precisely whent its length is one. More generally the length of a JavaScript string is its number of characters.

Anomalies:
The entities &lt;, &gt;, and &amp; and &nbsp;
With the four entities above as exceptions, any entity written in inner HTML will read back as a single character, namely the corresponding JavaScript character. However, if any of the above four entities (in named, decimal or hexidecimal form ) are written in innerHTML, they will be read back as the named entity as a string, not a single character.
  1. For example, &#x3C; and '\x3C' are the hexidecimal entity and JavaScript character for <,
    1. if you write <div> &#x3C; </div> and read back the innerHTM you get the named entity string "&lt;", of four charachters, and not the single character '<' or '\x3C'.
    2. Also if you write div.innerHTML = '\x3C', then when reading back the innerHTML you would expect the inner HTML to be the single character '\x3C', but again what you get is &lt;, a string of four characters.

The moral of the story is that if you map &lt;, &gt;, &amp; and &nbsp;, their equivalent decimal or hexidecimal entities, or their equivalent JavaSript characters '\x3C', '\x3E', '\x26', and '\x0A' to HTML, you get the named entities as strings, not the single JavaScript characters.

The non breaking space entity &nbsp;
The non breaking space &nbsp; does produce a web symbol, which is a thin amount of white space. However, it does more. It with its adjoining characters on its left and right will not be broken across lines in the web page.

The non breaking space has no editor symbol, i.e., there is no symbol that you can write in the editor to produce the non breaking space on the web page . If you copy the non breaking space's web symbol, then it is pasted into the editor as the normal space '\x20' obtained by striking the space bar. However, if you paste the web symbol into a content editable div you do get the non breaking space &nbsp;.

Also if you press the space bar inside a content editable div, it becomes the non breaking space &nbsp; in the source code.

Entities and Characters for the HTML Editor and outerHTML

White Spaces that Show up in outerHTML

The following characters.

  1. '\x0A' = = 'u000A' = '\n' (new line obtained by striking the enter key)
  2. '\x09' = '\u0009' = '\t' (horizontal tab obtained by striking the tab key)
  3. '\x20' = '\u0020' (normal space character obtained by striking the space bar)
are white spaces that show up in the editor and in the outerHTML of any containing HTMLElement. They are not condensed in the editor or outerHTMLs. They are condensed only in the display of the web page.

There is one exception. White space occurring inside a tag may or may not show up in the outerHTML depending upon the browser.

Thus one may parse an HTMLElement's outer HTML and write new HTML by substituting different characters for these white spaces. This is an important key in the author's JSHTML, which is bacically an editor for displaying JavaScript in HTML.

Entities that show up as white spaces in outerHTML

Writing the En space &ensp; entity in the editor shows up as the character '\x2002' in outerHTML, which is displayed as a space. Likwise copy and pastiong the En space web symbol into the editor has the same effect. The same discussion holds for the for the Em space &emsp; = '\x2003' and the thin space &thinsp; = '\u2009'.

Escaped Characters

The < symbol inside the editor is reserved for the start of tags. If you used this character outside this scenario, parsing the HTML, even if possible, would be a nightmare to determine whether < stands for a start tag or the less than symbol. Thus if not used as the start of a tag, the less than symbol must be escaped. &lt; is used for this purpose and produces the less than symbol on the web page.

Like wise in the editor the > symbol is used to terminate tagd and so must be escaped as &gt; to produce the greater than symbol on the web page.

Since the & character is used in the editor for writing HTML entities such as &Aacute;, parsing is possible but very inefficient to also use & to display & on the web page. Thus & is escaped as &amp; to efficiently display & on the web page.

Two Way Editor

The author has a two way HTML Editor written in HTML/JavaScript (A one way editor is simpler). A content editable pre element is used as the HTML editor, while an content editable div is used as the "web page". Changes to the source HTML in the pre are updated to the div web page. Conversely changes to the div web page are updated to the pre HTML editor. Since En, Em and thin spaces pasted into the the web page can display as white space characters in the editor, when the web page is parsed to obtain editor code, HTML entities can be substituted for the white spaces. This means readabiltiy, for otherwise there's no way to know how the spaces were obtained.
